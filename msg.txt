fix/feat: Correcao da Pesquisa de Propriedades + Sistema de Reservas

## Causa Raiz dos Problemas

### Problema 1: Propriedades nao aparecendo na tela inicial e pesquisa
- CAUSA: Os hooks useSearchProperties chamavam a API com datas fixas no passado
  (new Date(2026, 4, 10) = 10/Maio/2026), filtrando apenas vagas disponiveis
  NAQUELE periodo especifico.
- IDENTIFICACAO: Analise do codigo em hooks/api/property/useSearchProperties.ts
  e app/page.tsx mostrou que datePeriod era passado com valores hardcoded.
- CORRECAO: Removidas as datas hardcoded. A busca agora retorna todas as
  propriedades com vagas aprovadas sem filtrar por periodo de data.
- IMPACTO: Propriedades com vagas APROVADAS agora aparecem corretamente
  na tela inicial e na pagina de pesquisa.

### Problema 2: Tab de Reservas na conta exibia sempre vazio
- CAUSA: bookings.tab.tsx usava apenas um array estatico vazio como placeholder.
  Nenhuma chamada de API era feita.
- IDENTIFICACAO: Comentario no codigo "TODO: replace with real Booking type".
- CORRECAO: Implementado uso do hook useGetReservations() para buscar reservas
  reais do usuario autenticado via GET /reservations.
- IMPACTO: Tab "Reservas" na conta agora exibe reservas reais com status,
  datas, nome da vaga, propriedade e veiculo. Permite cancelar reservas
  PENDENTES e APROVADAS via PATCH /reservations/{id}/cancel.

### Problema 3: Datas de reserva hardcoded no fluxo de criacao
- CAUSA: handleReserve em spot/[id]/page.tsx usava datas fixas ("2026-01-01"
  e "2026-05-10") independente da disponibilidade real da vaga.
- IDENTIFICACAO: Analise direta do codigo de handleReserve.
- CORRECAO: O sistema agora le spot.operatingHours.datePeriod.start/end
  e usa essas datas ao criar a reserva via POST /reservations.
- IMPACTO: A reserva e criada com o periodo correto da vaga. Inputs de data
  editaveis foram substituidos por exibicao somente leitura do periodo e
  horario da vaga (startDate, endDate, timePeriod).

### Problema 4: Dashboard nao exibia reservas do locatario
- CAUSA: Dashboard so chamava useGetReservations(true) - apenas reservas
  recebidas como proprietario.
- CORRECAO: Adicionado segundo hook useGetReservations() (sem argumento)
  para buscar tambem as reservas feitas pelo proprio usuario como locatario.
- IMPACTO: Dashboard agora exibe secoes separadas: "Minhas Reservas"
  (como locatario) e secoes do proprietario (proximas reservas recebidas,
  historico, solicitacoes para aprovar/rejeitar).

### Correcao menor: loaded nunca retornava true em useGetReservations
- CAUSA: setLoaded(true) nunca era chamado apos carregar os dados.
- CORRECAO: Adicionado setLoaded(true) apos setReservations(data).

## Endpoints Utilizados
- GET /reservations - reservas do usuario autenticado (locatario)
- GET /reservations/owner - reservas recebidas pelo proprietario
- POST /reservations - criar nova reserva
- PATCH /reservations/{id}/cancel - cancelar reserva
- PATCH /reservations/{id}/approve - aprovar reserva (proprietario)
- PATCH /reservations/{id}/reject - rejeitar reserva (proprietario)
- GET /reservations/search/address - pesquisa de propriedades
